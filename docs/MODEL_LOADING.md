# Подробная инструкция по загрузке модели

Пошаговое руководство по загрузке модели в кластер Chaboss Cluster.

---

## Подготовка

1. **Запустите воркеры** на всех узлах кластера:
   ```bash
   python -m scripts.run_worker --config config/worker.yaml
   ```
2. **Запустите мастера** — из GUI (кнопка «Старт мастера» на вкладке «Настройки») или из терминала:
   ```bash
   python -m scripts.run_master
   ```
3. **Откройте GUI** (если ещё не открыт):
   ```bash
   python -m ui.main_window
   ```

---

## Шаг 1. Подключение к мастеру и конфиг воркеров

1. На вкладке **«Настройки»** в поле **«Подключение к мастеру»** укажите адрес мастера (например, `127.0.0.1:60051`).
2. В блоке **«Конфиг воркеров»** добавьте строки для каждого воркера: **IP (host)**, **Port**, при необходимости **Key (auth_token)**.
3. Нажмите **«Применить конфиг на мастере»** — мастер получит список воркеров и начнёт к ним подключаться.
4. Перейдите на вкладку **«Воркеры»** и нажмите **«Скан»** — в таблице появятся воркеры и их статусы (ONLINE/OFFLINE и т.д.). В логе отобразятся события подключения/отключения воркеров.

---

## Шаг 2. Проверка свободных ресурсов

1. После **«Скан»** на вкладке **«Настройки»** в блоке **«Режим загрузки модели»** появится строка:
   - **«Свободно в кластере: RAM X MB, VRAM Y MB. При выбранном Z% под модель влезает примерно до N GB»**  
   Это сумма свободной RAM и VRAM по всем воркерам и примерный размер модели, который влезет при выбранном проценте.
2. При необходимости измените **«Использование свободных ресурсов (%)»** (по умолчанию 75%). Чем выше процент, тем большая модель может быть загружена в режиме «модель влезает в кластер».

---

## Шаг 3. Выбор режима загрузки

В блоке **«Режим загрузки модели»** выберите режим:

- **«Модель влезает в кластер»** — модель целиком (в разделённом по воркерам виде) помещается в свободные ресурсы. Мастер разбивает модель по слоям и рассылает шарды воркерам.
- **«Модель по частям (стриминг)»** — для моделей, которые не влезают в доступные ресурсы; загрузка и вычисления по частям (режим в разработке).

Для большинства сценариев при достаточном количестве воркеров и VRAM выбирайте **«Модель влезает в кластер»**.

---

## Шаг 4. Загрузка модели

1. В поле **«Модель HuggingFace»** введите идентификатор модели (например, `bert-base-uncased`, `gpt2`, `Qwen/Qwen2-0.5B`).
2. Нажмите **«Старт»** — начнётся загрузка: мастер скачивает модель (или берёт из кэша), разбивает по слоям и рассылает шарды воркерам. Ход отображается в **логе** (нижняя часть вкладки «Воркеры»).
3. При успехе в статусной строке появится сообщение «Модель загружена и разослана воркерам», в логе — соответствующая запись.

---

## Шаг 5. Если загрузка не удалась

При ошибке загрузки появится диалог с текстом ошибки и рекомендациями:

- **Модель не влезает в выбранный % ресурсов** — увеличьте «Использование свободных ресурсов (%)» (например, до 80–90%).
- **Модель не влезает даже при 100%**:
  - **(a)** Переключитесь в режим **«Модель по частям (стриминг)»** (когда он будет реализован).
  - **(b)** Добавьте воркеров в кластер (в конфиг воркеров и «Применить конфиг на мастере»), чтобы увеличить суммарные RAM/VRAM.

После изменений повторите загрузку (**«Старт»**).

---

## Выгрузка модели

Чтобы освободить VRAM на воркерах, нажмите **«Выгрузить модель»**. Мастер отправит команду выгрузки всем воркерам; в логе появится подтверждение.

---

## Лог и статусы воркеров

- На вкладке **«Воркеры»** в нижней части окна отображается **лог** с метками времени: подключения/отключения воркеров, смена статуса, загрузка и выгрузка модели, ошибки.
- Подвижный разделитель между таблицей и логом позволяет изменить соотношение 70/30 по своему усмотрению.
- Кнопка **«Очистить лог»** очищает содержимое лога.

Используйте лог для диагностики: при отключении воркера в таблице обновится статус, а в логе появится запись вида «Воркер отключён: host:port» или «Воркер host:port: статус ONLINE → OFFLINE».

### Логи процесса воркера (файл)

Если воркер запущен из GUI (**«Старт воркера»**), его **stdout/stderr отключены**, поэтому в консоли ничего не видно. Все сообщения воркера дублируются в файл:

**`logs/worker.log`** (в корне проекта, рядом с `config/`).

Откройте этот файл на машине воркера (в т.ч. на Windows), чтобы увидеть этапы `InitShard`, загрузку HF и ошибки.

## Qwen3.5 MoE и GPTQ (например Qwen3.5-35B-A3B-GPTQ-Int4)

У таких моделей веса лежат под префиксом **`model.language_model.layers.N.*`**, а не `model.layers.*`. Раньше мастер **не распознавал** это и делил веса **round-robin по ключам**: оба воркера вытягивали тензоры из **каждого** файла safetensors, из‑за чего на Windows рос пик виртуальной памяти и часто возникала ошибка **1455 (файл подкачки)**.

Сейчас мастер режет шарды **по слоям** (`model.language_model.layers`), и каждый воркер читает только **свои** слои — лишние файлы только сканируются по списку ключей без загрузки тензоров.

Чекпоинты **GPTQ** (ключи `qweight`, `qzeros`, …) **нельзя** собрать в обычный `nn.Module` на воркере как у не-квантованной модели. Для стабильной работы на слабом узле используйте режим **«Модель по частям (стриминг)»** и при необходимости увеличьте файл подкачки Windows.
